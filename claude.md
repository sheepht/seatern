> Product Requirements Document  
> 版本：1.2
> 日期：2026-02-05

---

## 1. 產品概述

### 1.1 產品定位

一套整合「賓客資料收集」與「智慧座位安排」的系統，透過 Community Detection 演算法與滿意度評分機制，協助新人高效完成婚禮、尾牙、活動的座位規劃。

### 1.2 目標用戶

| 用戶類型 | 使用場景 |
|---------|---------|
| 新人 | 婚禮座位安排 |
| HR / 行政人員 | 尾牙、公司活動座位安排 |
| 活動企劃 | 各類宴會座位規劃 |

### 1.3 核心價值主張

- 從賓客填表到座位輸出，一條龍整合
- 視覺化社交關係圖，一眼掌握人際結構
- 量化滿意度分數，每次調整都有數據支撐
- 即時回饋拖曳影響，避免拆東牆補西牆

## 3. 功能規格

### 3.1 新人端：賓客清單管理

#### 3.1.1 賓客資料欄位

| 欄位 | 必填 | 說明 |
|-----|------|-----|
| 主要姓名 | ✓ | 正式顯示名稱 |
| 別名/暱稱 | | 支援多個，用於賓客搜尋配對 |
| 分類（category） | | 主分類，由活動定義可用選項（如婚禮：男方/女方/共同） |
| 關係分數 | ✓ | 1-3 分，代表與主辦人親疏程度 |
| 子分類 | | 單選：大學同學、公司同事、家人、自訂...（隸屬於分類）|
| 避免同桌 | | 新人標記，賓客不可見 |

#### 3.1.2 別名設計目的

同一人可能被不同賓客以不同稱呼選擇：

```
主要姓名：陳志明
別名：小明、阿明、明哥、David
```

賓客在表單選「想同桌」時，搜尋會同時比對所有別名。

#### 3.1.3 「避免同桌」標記

此功能僅限新人後台使用，不出現在賓客表單：

```
新人端操作：
├── 點擊任意兩位賓客
├── 標記為「避免同桌」
└── 原因（選填）：前任 / 家庭糾紛 / 工作嫌隙 / 其他
```

系統處理邏輯：
- 不影響滿意度計算
- 作為排位硬約束
- 若被排在同桌，顯示違規警告而非扣分

### 3.2 賓客端：填寫表單

#### 3.2.1 表單欄位

| 欄位 | 必填 | 說明 |
|-----|------|-----|
| 確認出席 | ✓ | 確認 / 婉拒 |
| 成人人數 | ✓ | 出席成人數（含本人，1-2，預設 1） |
| 嬰兒人數 | | 需要嬰兒椅的數量（預設 0） |
| 想同桌的人 | | 最多選 3 位，可排優先順序 |
| 飲食需求 | | 素食 / 過敏 / 不吃牛 / 其他 |
| 特殊需求 | | 輪椅 / 兒童椅 / 靠近出口 / 其他 |

#### 3.2.2 「想同桌」選擇 UX

```
輸入框行為：
1. 賓客輸入文字
2. 模糊搜尋已建立的賓客名單（含別名）
3. 顯示下拉選項供選擇
4. 若找不到，可手打姓名（系統稍後嘗試配對）
```

#### 3.2.3 表單狀態管理

| 狀態 | 說明 |
|-----|------|
| 未發送 | 連結尚未寄出 |
| 待填寫 | 已發送，賓客尚未填 |
| 已填寫 | 賓客已提交 |
| 已修改 | 賓客修改過（記錄修改時間） |
| 已取消 | 賓客表示不出席 |

### 3.3 Community Detection 圖

#### 3.3.1 圖的組成元素

```
節點（Node）= 賓客
├── 大小 = 與新人關係分數（越大越重要）
├── 顏色 = 來源（藍=男方、紅=女方、紫=共同）
└── 位置 = 由演算法自動佈局

邊（Edge）= 關係連結
├── 粗線 = 雙向都選「想同桌」
├── 細線 = 單向選擇或同群組
└── 虛線 = 系統推測的關係

社群（Community）= 虛線圈
├── 由演算法自動偵測
├── 代表緊密連結的群體
└── 通常對應到一個群組或混合群體
```

#### 3.3.2 視覺化範例

```
        ┌───────────────────┐
        │    社群 A          │
        │   ┌───┐           │
        │   │ 1 │◄──┐       │
        │   └───┘   │       │
        │     │     │       │
        │     ▼     │       │
        │   ┌───┐ ┌───┐     │
        │   │ 2 │─│ 3 │     │
        │   └───┘ └───┘     │
        └───────────────────┘
              │
              ▼
            ┌───┐
            │ 4 │ ← 橋接者：連接多個社群
            └───┘
              │
    ┌─────────┴─────────┐
    ▼                   ▼
┌───────┐           ┌───────┐
│ 社群 B │           │ 社群 C │
│ (5人)  │           │ (3人)  │
└───────┘           └───────┘
```

#### 3.3.3 演算法選擇

| 演算法 | 特性 | 適用場景 |
|-------|------|---------|
| Louvain | 效果好，速度適中 | 最終排位計算 |
| Label Propagation | 速度最快 | 即時互動預覽 |

#### 3.3.4 互動功能

| 操作 | 效果 |
|-----|------|
| 點擊社群 | 顯示成員名單與統計 |
| 點擊節點 | 顯示賓客詳細資料與滿意分數 |
| 拖曳社群 | 整組指派到某一桌 |
| 圈選多節點 | 批次設為同一桌 |
| 懸停邊 | 顯示關係類型與強度 |

### 3.4 滿意度計算系統

#### 3.4.1 計算公式

```
個人滿意度 = 基礎分 + 群組分 + 偏好分 + 需求分

滿分：100 分
基礎分：50 分
```

#### 3.4.2 群組分（最高 +20 分）

來自新人的初始分類，反映「跟熟人坐一起」的程度：

| 條件 | 加分 |
|-----|------|
| 同桌有同群組的人佔 50% 以上 | +20 |
| 同桌有同群組的人佔 30-50% | +15 |
| 同桌有同群組的人佔 10-30% | +10 |
| 同桌只有 1 位同群組 | +5 |
| 完全沒有同群組的人 | +0 |

#### 3.4.3 偏好分（最高 +25 分）

來自賓客填的「想同桌」，反映個人意願被滿足的程度：

| 條件 | 加分 |
|-----|------|
| 3 位想同桌的人都在同桌 | +25 |
| 2 位在同桌 | +18 |
| 1 位在同桌 | +10 |
| 0 位在同桌，但有人在鄰桌 | +5 |
| 完全沒配到 | +0 |

#### 3.4.4 需求分（固定 +5 分）

所有賓客一律 +5 分。飲食與特殊需求以備註欄記錄，供主辦人參考，不影響滿意度計算。

#### 3.4.5 分數解讀

| 分數區間 | 狀態 | 顯示顏色 |
|---------|------|---------|
| 85-100 | 被照顧得很好 | 🟢 綠色 |
| 70-84 | 安排合理 | 🟡 黃色 |
| 55-69 | 普通，有優化空間 | 🟠 橘色 |
| < 55 | 需要關注 | 🔴 紅色 |

#### 3.4.6 特殊情況處理

**賓客尚未填表時：**
- 只有群組分，沒有偏好分
- 滿意度上限為 75 分
- 顯示標記：`(暫定)`

**溢出賓客補償：**

當同群組人數超過單桌上限，部分人被安排到鄰桌：

| 條件 | 補償加分 |
|-----|---------|
| 被標記為「溢出安排」 | +3 |
| 同群組在鄰桌（距離 ≤ 1 桌） | +5 |

### 3.7 群組溢出處理

#### 3.7.1 情境說明

當同一群組人數超過單桌上限時：

```
範例：
├── 高中同學群：12 人
├── 每桌上限：10 人
└── 溢出：2 人需要安排到其他桌
```

#### 3.7.2 拆分策略選項

```
當群組人數超過單桌上限時：

○ 盡量集中（預設）
   → 主桌塞滿 10 人，溢出 2 人併到鄰桌
   → 適合：想讓大部分人坐一起
   
○ 平均分配
   → 分成 6+6 兩桌，各自再補其他人
   → 適合：群內有小圈圈，可自然分開
   
○ 手動決定
   → 新人自己選誰溢出
```

#### 3.7.3 溢出者選擇邏輯

系統自動建議優先溢出：

1. 該圈子中與新人關係分數較低的人
2. 有其他社交連結的人（例如同時是高中同學又是公司同事）
3. 主動填了「也想跟其他圈子的人坐」的人

#### 3.7.4 鄰桌安排


```typescript
  type: EventType                 // 活動類型（預設 wedding）
```

```typescript
interface Contact {
  id: string

  aliases: string[]               // 別名列表
  email?: string                  // 聯絡 email（發送表單用）

}
```

### 4.4 賓客（Guest）

```typescript
interface Guest {
  id: string
  eventId: string                 // 所屬活動
  contactId: string               // 連結通訊錄（姓名、別名從 Contact 讀取）
  category?: string               // 主分類（婚禮：男方/女方/共同；企業：部門名等）
  relationScore: number           // 1-3，與主辦人親疏（每次活動可不同）
  subcategoryId?: string          // 所屬子分類 ID

  // 賓客填寫
  rsvpStatus: 'pending' | 'confirmed' | 'declined' | 'modified'
  attendeeCount: number           // 成人出席人數（含本人，1-2，預設 1）
  infantCount: number             // 嬰兒人數（預設 0）
  dietaryNote?: string            // 飲食需求備註（素食、過敏等）
  specialNote?: string            // 特殊需求備註（輪椅、兒童椅等）

  // 系統計算
  satisfactionScore: number       // 滿意度分數
  assignedTableId?: string        // 分配的桌次 ID
  isOverflow: boolean             // 是否為溢出安排
  isIsolated: boolean             // 是否為孤立賓客
}
```

### 4.5 關係邊（Edge）

```typescript
interface Edge {
  from: string                    // 賓客 ID
  to: string                      // 賓客 ID
  weight: number                  // 關係強度
  type: 'mutual' | 'one-way' | 'same-group' | 'inferred'
  // mutual: 雙向都選想同桌
  // one-way: 單向選擇
  // same-group: 同群組自動建立
  // inferred: 系統推測
}
```

```typescript
interface Subcategory {

}
```


### 4.8 桌次（Table）

```typescript
interface Table {
  id: string
  eventId: string                 // 所屬活動
  name: string                    // 桌名（可自訂）
  capacity: number                // 容量上限（預設 10）
  guestIds: string[]              // 已安排的賓客 ID
  positionX: number               // 畫布 X 座標（自由拖曳定位）
  positionY: number               // 畫布 Y 座標（歐幾里德距離算鄰桌）
  averageSatisfaction: number     // 平均滿意度
  color?: string                  // 桌次顏色
  note?: string                   // 備註
}
```

### 4.9 座位快照（SeatingSnapshot）

```typescript
interface SeatingSnapshot {
  id: string
  eventId: string                 // 所屬活動（每活動最多一份）
  name: string                    // 快照名稱
  data: {                         // 快照資料
    guests: Array<{ guestId: string, tableId: string, satisfactionScore: number, isOverflow: boolean }>
    tables: Array<{ tableId: string, positionX: number, positionY: number }>

  averageSatisfaction: number     // 快照時的全場平均滿意度
}
```

新人按「儲存排位」時拍快照到 SeatingSnapshot，不滿意新排位可一鍵回復。

## 7. 附錄

### 7.1 名詞定義

| 名詞 | 定義 |
|-----|------|
| 滿意度 | 量化指標，代表賓客對座位安排的滿意程度（0-100） |
| 群組分 | 滿意度組成之一，來自「同桌有同群組的人」 |
| 偏好分 | 滿意度組成之一，來自「想同桌的人有配對成功」 |
| 溢出 | 同群組人數超過單桌上限時，部分人被安排到其他桌 |
| 孤立賓客 | 社交連結薄弱，需要特別關照的賓客 |
| 硬約束 | 必須遵守的規則，違反會顯示警告（如避免同桌） |
| 軟約束 | 盡量滿足的偏好，影響滿意度分數但不強制 |

### 7.2 滿意度公式速查

```
滿意度 = 50（基礎）+ 群組分（0-20）+ 偏好分（0-25）+ 需求分（0-5）

群組分：
  50%+ 同群組 → +20
  30-50%     → +15
  10-30%     → +10
  僅 1 人     → +5
  無          → +0

偏好分：
  3/3 配對   → +25
  2/3 配對   → +18
  1/3 配對   → +10
  0 但鄰桌有 → +5
  完全沒有   → +0

需求分：
  一律 +5（備註欄記錄，不影響分數）
```

## 開發注意事項

### 跨側欄與畫布的視覺元素
側欄是 HTML（z-10），畫布是 SVG（overflow: visible）。任何需要從側欄賓客 chip 連線到畫布桌子的視覺元素（推薦虛線、動畫等），**必須**用 `createPortal(..., document.body)` 渲染到 `<body>`，不能放在 FloorPlan 的 SVG 或其父容器內，否則會被側欄的 stacking context 擋住。

規則：
1. 用 `createPortal(..., document.body)` 渲染到 body
2. 用 `position: fixed` + 高 z-index（9998）
3. 用螢幕座標（getBoundingClientRect + getScreenCTM）而非 SVG 座標
4. 設 `pointer-events: none`
