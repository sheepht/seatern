### 非活躍 Event / User 資料清理策略 (P2)
- **什麼：** 定期清理長期未活躍的 Event（以及連帶的 auth.users），避免 SaaS 資料累積
- **為什麼：** 訪客玩一下就走會留下孤兒 Event；長期累積既占空間也可能有個資法合規風險（目的消失應主動刪除）
- **前提：** E2E 測試已落地（可作為 regression 安全網，驗證清理不誤殺活躍資料）
- **設計要點：**
  - 「未活躍」定義：候選有「從未新增 Guest 的空殼」/「N 天沒被開啟」/「N 天沒被修改」，三者抓到的人群不一樣
  - **資料遺失風險大**：新人可能提前 10 個月建活動、等婚禮前才回來，cron 不能無預警砍
  - 必須有預警機制：刪除前 N 天寄信「活動將在 X 天後刪除，登入即可延續」
  - 隱私權條款要寫明資料保留政策，合規面也要對
  - user row 本身只有 1-2 KB，真正的空間大頭是 Event + Guest + Table + Snapshot，清 user 省的空間極小
- **Effort：** M（human）→ S（CC+gstack）
- **來源：** 2026-04-11 規劃 E2E 測試時發現，從測試資料清理延伸出產品需求
